📅 Scheduling Transfer Workflow (IMMEDIATE)

flowchart TD subgraph SCHEDULE["📅 SCHEDULING REQUEST"] A[/"Caller Requests Scheduling"/] --> B["Detect Scheduling Intent"] end subgraph IMMEDIATE["⚡ IMMEDIATE TRANSFER"] B --> C["Do NOT collect additional data"] C --> D["Trigger Transfer Tool Immediately"] end subgraph COMPLETE["✅ TRANSFER COMPLETE"] D --> E[/"Call Transferred to Scheduling Team"/] E --> F["Human Agent Handles Data Collection"] end

⚡ IMMEDIATE TRANSFER — No Data Collection:

  • When caller requests scheduling, immediately trigger the transfer tool
  • Do NOT collect patient name, DOB, or imaging service
  • The human scheduling team will gather all required information
  • Universal variables (caller_name, caller_phone) are already captured in Steps 1-2

📝 Variables Already Available (from Steps 1-2):

  • {{caller_name}} — Caller's full name
  • {{caller_phone}} — Caller's phone number

🧑‍💼 Live Operator Transfer (Last Resort)

⚠️ CRITICAL: Last Resort Only

  • The Live Operator transfer is a LAST RESORT option
  • Agent must exhaust all other options before offering this transfer
  • Use for: complaints, escalations, complex situations, non-imaging requests

When to Use

Scenario Action
Caller asks for "live person" at start of call Do NOT transfer — ask what they need and attempt to assist
Caller has complaint or escalation Transfer to Receptionist after attempting to understand
Complex situation outside standard workflows Transfer to Receptionist after clarifying questions
Caller confused/frustrated after multiple attempts Transfer to Receptionist
Non-imaging request (vendor, job inquiry) Transfer to Receptionist

🔄 Multi-Intent Call Example

Scenario: Caller needs both medical records AND has a billing question

  1. Records Workflow: Agent collects records_* variables
  2. Agent asks: "Anything else I can help with?"
  3. Caller mentions billing question
  4. Billing Workflow: Agent collects billing_* variables
  5. Agent asks: "Anything else I can help with?"
  6. Caller says no
  7. Agent collects email and closes call

Final {{intents_handled}}: "records, billing"

This allows downstream webhooks to route emails based on which intents were handled.

📊 Data Collection Requirements

Variable Convention: Intent-specific variables use prefixes (records_, billing_) to support multi-intent calls.

Universal Variables (All Calls)

Variable Field Data Type Notes
{{caller_name}} Caller Name string Full name of person calling (Step 1)
{{caller_phone}} Caller Phone string Best contact number (Step 2)
{{caller_email}} Caller Email string Collected before closing
{{caller_reason}} Reason for Call string Initial stated reason (Step 3)
{{intents_handled}} Intents Handled string (comma-separated) e.g., "records, billing" — tracks completed workflows

Scheduling Request (IMMEDIATE TRANSFER)

Variable Field Data Type Notes
⚡ NO ADDITIONAL DATA COLLECTED — Immediate transfer to scheduling team

Billing Inquiry (billing_* prefix)

Variable Field Data Type Notes
{{billing_patient_name}} Patient Name string Only if different from caller
{{billing_patient_dob}} Patient DOB string (YYYY-MM-DD) Patient's date of birth
{{billing_reason}} Reason for Call string Specific billing inquiry details

Medical Records Request (records_* prefix)

Variable Field Data Type Notes
{{records_patient_name}} Patient Name string Only if different from caller
{{records_patient_dob}} Patient DOB string (YYYY-MM-DD) Patient's date of birth
{{records_imaging_service}} Imaging Service string Type of imaging performed
{{records_imaging_dos}} Imaging DOS string (YYYY-MM-DD) Date of service
{{records_delivery_method}} Delivery Method enum "email" or "fax"
{{records_delivery_destination}} Email/Fax string Where to send records
{{records_caller_type}} Caller Type enum Relationship to patient
{{records_release_required}} Release Required string "YES" or "NO" — Auto-set based on caller_type
