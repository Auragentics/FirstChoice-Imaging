1️⃣
Workflow 1: Contact DB UpdateWebhook-triggered. Creates/updates contact with all 16 extracted variables. Fires immediately when call ends.
2️⃣
Workflow 2: Post-Transcript ActionsTriggered by Generate Transcript. Routes by intents_handled. Sends staff emails + caller SMS based on intent, caller type, and request type.

Why Two Workflows?

  • Workflow 1 fires on the webhook POST and handles only contact data — fast, reliable, no dependencies
  • Workflow 2 fires after the transcript is generated and uses the contact variables to route SMS and emails — ensures transcript is available for staff email attachments
  • Separation keeps each workflow focused and easier to debug

Workflow 1 Summary

  • Trigger: Inbound Webhook (POST from Retell AI)
  • Action 1: Create or Update Contact — lookup by phone number
  • Action 2: Map all 16 webhook fields to GHL custom contact fields
  • Action 3: Add tag "AI Voice Agent Inquiry" for reporting/filtering
  • No conditional logic — every field is mapped regardless of intent

Critical: Use "Contains" (Not "Equals")

  • Multi-intent calls produce comma-separated values like "records,billing"
  • Each If/Else branch checks if intents_handled contains the value
  • This means a call with "records,billing" triggers BOTH Branch 1 and Branch 2
  • All 3 branches are evaluated independently — they are NOT mutually exclusive

Workflow 2 Summary

  • Trigger: Generate Transcript action completes
  • Gate: Skip if intents_handled is empty (scheduling/FAQ calls)
  • Branch 1 (Records): Staff email + SMS with form link (varies by caller_type & request_type)
  • Branch 2 (Billing): Staff email + confirmation SMS
  • Branch 3 (Lien): Staff email + confirmation SMS
  • SMS Gate: Every SMS is gated by sms_consent = YES

Records SMS Decision Matrix

Caller Type Request Type SMS Content Form Link?
patient medical_records Medical records release form link Yes
patient radiology_report Radiology report request form link Yes
provider medical_records Provider records transfer form link Yes
provider radiology_report Provider radiology report form link Yes

GHL Implementation: Nested If/Else

  • Outer If/Else: Check records_caller_type (patient vs provider vs attorney)
  • Inner If/Else (patient branch): Check records_request_type (medical_records vs radiology_report)
  • Inner If/Else (provider branch): Check records_request_type (medical_records vs radiology_report)
  • Attorney branch: No nested check — always sends confirmation-only SMS
  • Each leaf node uses a different SMS template with the appropriate form link

SMS Consent is REQUIRED

  • sms_consent must equal "YES" for ANY SMS to be sent
  • If sms_consent = "NO", the entire SMS branch is skipped
  • Staff emails are ALWAYS sent regardless of SMS consent
  • This ensures A2P 10DLC compliance — no unsolicited texts

Multi-Intent SMS Example

A caller with intents_handled = "records,billing" and sms_consent = YES receives:

  • SMS 1: Records form link (based on their caller_type + request_type)
  • SMS 2: Billing confirmation ("Your billing inquiry has been received...")

These can be sent as separate messages or combined into a single SMS depending on your GHL SMS action configuration.
