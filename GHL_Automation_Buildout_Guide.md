## WORKFLOW 1: Contact Database Update

**Purpose:** Receives the webhook POST from Retell AI when a call ends, creates or updates the contact in GHL with all extracted variables.

**Total actions: 3**

---

### WF1 — Step 1: Trigger

**Action Type:** `Inbound Webhook`

| Setting | Value |
|---------|-------|
| Trigger Name | Retell AI Post-Call Webhook |
| Webhook URL | *(auto-generated by GHL — copy this URL)* |

**After adding the trigger:**
1. Copy the GHL-generated webhook URL
2. Paste it into Retell AI's Post-Call Webhook configuration
3. Fire a test call from Retell so GHL detects the incoming fields
4. GHL will display the payload fields for mapping in subsequent actions

## WORKFLOW 2: Post-Transcript Actions

**Purpose:** After the transcript is generated, routes SMS and email notifications based on `intents_handled`, `records_caller_type`, `records_request_type`, and `sms_consent`.

**Total actions: ~25 (1 trigger + 1 gate + 3 intent branches with nested logic)**

---

### WF2 — Step 1: Trigger

**Action Type:** `Generate Transcript` *(trigger fires when transcript generation completes)*

| Setting | Value |
|---------|-------|
| Trigger Name | Post-Transcript Actions |

> This ensures the call transcript is available for staff emails.

---

### WF2 — Step 2: Gate Check — intents_handled is not empty

**Action Type:** `If/Else`

| Setting | Value |
|---------|-------|
| Action Name | Gate: Has Intents? |
| Condition | `Contact > Intents Handled` **is not empty** |

```
IF  intents_handled  IS NOT EMPTY
├── YES → Continue to Step 3 (intent branches)
└── NO  → STOP (no further actions — this was a scheduling/FAQ call)
```

> **Why this gate?** Scheduling calls and general FAQ calls produce `intents_handled = ""`. Without this gate, the workflow would run all If/Else branches on every call and find nothing to do. The gate exits early.

---

### WF2 — Step 3: Branch 1 — Records

*Place all of Steps 3–10 inside the YES path of Step 2.*

**Action Type:** `If/Else`

| Setting | Value |
|---------|-------|
| Action Name | Branch 1: Records? |
| Condition | `Contact > Intents Handled` **contains** `records` |

```
IF  intents_handled  CONTAINS  "records"
├── YES → Continue to Step 4 (Records Email)
└── NO  → Skip to Step 11 (Branch 2: Billing)
```

> **CRITICAL:** Use **contains**, not **equals**. Multi-intent calls produce values like `"records,billing"` — "equals" would miss this.

## Key Design Rules

1. **Use CONTAINS for intents_handled** — never EQUALS. Multi-intent calls produce `"records,billing"`.
2. **Use EQUALS for all other fields** — `sms_consent`, `records_caller_type`, `records_request_type` are single-value enums.
3. **All 3 intent branches are independent** — they are NOT if/else-if/else. Each branch is its own If/Else at the same nesting level. A call with `"records,billing"` fires both Branch 1 and Branch 2.
4. **Staff emails ALWAYS send** — they are NOT gated by `sms_consent`. Only caller SMS is gated.
5. **SMS gate is per-branch** — each branch has its own `sms_consent = YES` check rather than one global check. This keeps the logic clean if a branch is added or removed later.
6. **Every SMS includes `Reply STOP to opt out`** — required for A2P 10DLC compliance.
7. **Form link placeholders** — Replace `[MEDICAL_RECORDS_FORM_LINK]`, `[RADIOLOGY_REPORT_FORM_LINK]`, `[PROVIDER_RECORDS_FORM_LINK]`, and `[PROVIDER_RADIOLOGY_FORM_LINK]` with actual First Choice Imaging URLs when available.
